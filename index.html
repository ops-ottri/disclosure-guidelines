OTR-08 ‚Äî eCBSV Architecture Decision ‚ö†Ô∏è Requires Elavon confirmation

Critical Finding from API Analysis USB Avvance's /pre-approval/application endpoint runs eCBSV as a backend verification during decisioning. There is no separate eCBSV API call ‚Äî USB uses the SSN, name, and DOB from the consumer object to verify against SSA records as part of generating offers. The /book endpoint accepts only a generic "consent": "yes/no" boolean.

Consequence: eCBSV consent must be collected before Ottri calls USB's application endpoint. Post-offer placement is not viable ‚Äî offers don't exist until after eCBSV check completes. Two options remain. Ask Elavon which is acceptable.

How it works

eCBSV consent is collected as item (2) inside the unified OTR-09 consent, using lender-agnostic language. When the waterfall routes to USB, consent is already obtained. When it routes to a non-USB lender, consent was collected but never exercised (no harm ‚Äî similar to collecting credit consent for all lenders pre-submission).

(2) SSN Verification. I authorize the Social Security Administration (SSA) to verify and disclose to the applicable lending institution, through its designated service provider, whether my name, Social Security Number (SSN), and date of birth match information in SSA records. This authorization is for a one-time validation within 90 days of submission.

Pros

Zero conversion impact ‚Äî no additional screen or step. Single consent gate. Simpler architecture (no mid-waterfall interstitial). OTR-09 becomes a 7-item consent that covers all Elavon ¬ß5.c requirements in one block.

Cons / Risk

SSA's eCBSV program may require the consent to name the specific requesting institution. USB's required text says "U.S. Bank through Early Warning Services, LLC" ‚Äî the generic "applicable lending institution" may not satisfy SSA program requirements. However: Elavon ¬ß5.c bullet 3 itself uses generic language ("authorization from Customer to the social security administration to verify‚Ä¶") without naming USB, suggesting Elavon may accept this framing.

Question for Elavon Does "the applicable lending institution and its designated service provider" satisfy eCBSV consent requirements for the SSA program? Or must the consent specifically name "U.S. Bank through Early Warning Services, LLC"?

How it works

Consumer submits OTR-09 (6 items, no eCBSV). Ottri runs soft pull. Decision engine determines USB is the target lender. Before calling USB's /pre-approval/application endpoint, Ottri presents a brief interstitial naming U.S. Bank specifically. Consumer agrees. Ottri calls USB. If USB declines and waterfall falls through to next lender, the consumer saw USB consent for nothing ‚Äî but no compliance issue.

Architecture requirement

Ottri's waterfall must be able to predict USB routing before calling the USB API. This means the routing decision (based on credit score, loan amount, eligibility criteria from Elavon ¬ß4.2) must happen in Ottri's engine before the API call, not as part of it. If the waterfall is "fire and see what comes back," this option doesn't work.

My consent is for a one-time validation within the next 90 days.

Pros

Perfect SSA compliance ‚Äî names the exact institution as required. Consumers only see USB consent when actually being routed to USB. No consent collected for relationships that don't materialize. Cleanest from a regulatory purist perspective.

Cons / Risk

Adds a conversion-killing step mid-flow. Consumer has already submitted, is waiting for results, and now gets a consent screen they didn't expect. If USB declines and Ottri falls through to next lender, the consumer saw USB consent for nothing (confusing, though not a compliance issue). Requires Ottri's routing engine to make deterministic routing decisions before API calls. More complex engineering.
